# Правила для AI агента (DeepSeek Flash)

## Роль
Ты — эксперт по Rust и чистой архитектуре. Следуешь строгим правилам генерации кода. Придерживаешься инструкций описанных в этом файле. Отвечаешь только на русском. Коммит и пуш только по запросу от пользователя.

## Проект
- Мы создаём приложение для форматирования программ G-кода для станков с ЧПУ с графическим интерфейсом.
- Программа умеет нумеровать кадры, расставлять пробелы между кодами, сохраняя целостность и последовательность команд. В коде могут встречаться специфические для станка команды — которые не должны изменяться.
- GUI реализован на egui (eframe), архитектура MVI.

## Структура проекта (Clean Architecture + Data Layer)
- `src/domain/` — сущности и бизнес-логика (не зависит от фреймворков)
- `src/application/` — use cases / интеракторы (парсер, форматтер, валидатор)
- `src/infrastructure/` — внешние зависимости (лексер, IO)
- `src/interfaces/` — GUI-обработчики (egui MVI)
- `src/data_layer/` — отдельный поток для ресурсоёмких операций (pipeline, IO)
- `src/shared/` — утилиты, конфиги, error types

### Правила зависимости между слоями
- **domain** → не зависит ни от чего.
- **application** → зависит только от `domain` и `shared`.
- **infrastructure** → зависит только от `domain` (реализует трейты).
- **interfaces** → зависит от `application`, `infrastructure`, `shared`, `data_layer`. **Не зависит** от `domain` напрямую (доступ к domain только через application).
- **data_layer** → зависит от `application`, `infrastructure`, `shared`. Импортирует `domain` только через `application`. Не зависит от `interfaces`.
- **shared** → не зависит ни от чего.
- Ни один слой не должен импортировать вышестоящий слой.

## Правила размещения файлов
- Один файл = одна логическая единица (модуль).
- Имена файлов: `snake_case.rs`.
- Модуль domain: только `struct`, `enum`, `trait` без реализации внешних эффектов.
- Модуль infrastructure: реализует трейты из domain (например `DefaultLexer` реализует `Lexer`).
- Тесты: unit-тесты в `*_tests.rs` рядом с кодом (через `#[path = "..."]`).
  Интеграционные GUI-тесты через `egui_kittest` — в `src/interfaces/gui/gui_tests.rs`.

## Требования к коду
- Rust 2021 edition, следуй clippy.
- Комментарии только на русском, но код и идентификаторы на английском.
- `thiserror` — для типов ошибок в `shared`.
- Избегай `unwrap()` — используй `?` и обработку ошибок. `unwrap()` допустим только в тестах, в production коде — нет.
- Для кода внутри `src/` используй `crate::` для обращения к другим модулям проекта, а не `code_parser::`. Исключение: бинарные крейты (`src/bin/`) — они обязаны обращаться по имени крейта (`code_parser::`), т.к. это отдельные entry points.

## Data Layer (отдельный поток)
- `src/data_layer/mod.rs` — определения `EditorCommand`, `EditorEvent` и функция `spawn_data_layer()`.
- `src/data_layer/pipeline.rs` — цепочка: лексер → парсер → валидатор → форматтер.

### Архитектура двух потоков
```
main thread (egui UI)
    │
    ├── std::sync::mpsc::Sender<EditorCommand>
    │         команды: Format, Validate, OpenFile, SaveFile...
    │         ▼
    └── std::sync::mpsc::Receiver<EditorEvent>
              события: Formatted, Validated, NotifyUser, Idle...
              ▲
         data thread (std::thread::spawn)
              CPU: лексер → парсер → валидатор → форматтер
              IO:  чтение/запись файлов
```

### Правила data layer
- **Tokio запрещён** — только `std::thread::spawn` и `std::sync::mpsc`.
- Data thread запускается один раз в `src/bin/gui_main.rs()` и живёт весь цикл приложения.
- Все CPU-интенсивные вычисления (лексер, парсер, форматтер, валидатор) выполняются в data thread.
- Всё синхронное чтение/запись файлов — в data thread.
- **Coalesce TextChanged**: если в канале несколько `TextChanged` подряд — обрабатывается только последний.
- **Debounce**: UI отправляет `TextChanged` не чаще раза в 100 мс.
- **Блокировка UI**: пока data thread занят, кнопки Format/Validate/Open/Save блокируются флагом `is_busy`.
- **Файловые диалоги** (`open`, `save as`) — синхронные (`rfd::FileDialog`), вызываются из UI потока по запросу data layer через `EditorEvent::RequestFilePicker`.
- Выход: при закрытии окна data thread завершается автоматически (канал закрывается).

### EditorCommand (команды от UI к data layer)
Вложенные перечисления, группировка по concern'ам:
- `Pipeline(PipelineCommand)`:
  - `TextChanged(String)` — текст изменился (для дебаунса).
  - `Format { content, renumber_step, skip_empty_lines }` — отформатировать.
  - `Validate(String)` — проверить.
- `File(FileCommand)`:
  - `OpenFile` — открыть файл.
  - `SaveFile { path: Option<String>, content: String }` — сохранить файл.
- `Dialog(DialogCommand)`:
  - `FilePickerResult { path, mode }` — результат выбора файла из диалога.
  - `DialogResult { confirmed }` — результат диалога подтверждения.

### EditorEvent (события от data layer к UI)
Вложенные перечисления, группировка по concern'ам:
- `Pipeline(PipelineEvent)`:
  - `Formatted { content, errors }` — результат форматирования.
  - `Validated { errors }` — результат проверки.
- `File(FileEvent)`:
  - `Loaded { content, file_path }` — файл загружен.
  - `Saved { file_path }` — файл сохранён.
- `Dialog(DialogEvent)`:
  - `RequestFilePicker { mode }` — запрос на открытие диалога выбора файла.
  - `RequestDialog { title, message }` — запрос на подтверждение.
  - `NotifyUser { message, level }` — уведомление пользователя.
  - `Idle` — data layer завершил обработку.

## Графический интерфейс (GUI)
- GUI располагается в `src/interfaces/gui/`.
- Архитектура: **MVI (Model-View-Intent)**.
  - `model/` — состояние приложения. Поля — приватные, доступ через геттеры (`.content()`, `.is_busy()` и т.д.),
    мутация — через `apply(intent)` или `pub(crate)` сеттеры (`.set_content()`, `.set_error_lines()` и т.д.).
  - `intent/` — намерения пользователя (enum).
  - `update/` — редьюсер: чистая функция, мутирует model на основе intent.
  - `view/` — отрисовка UI, возвращает intents.
  - `app/` — точка входа фреймворка, соединяет слои.
- Фреймворк UI — **immediate mode** (все виджеты пересоздаются каждый кадр).
- `TextEdit` (редактор кода) — исключение из MVI: мутирует локальную копию, после кадра
  изменения сохраняются в модель через `set_content()`.
- **Файловые диалоги** (`open`, `save as`) — синхронные, вызываются из UI потока по запросу data layer.
  Защита от повторного вызова через флаг блокировки.
- Все кнопки, инициирующие длительные операции, блокируются флагом `is_busy` на время выполнения.
- GUI поток **не должен** выполнять CPU-интенсивную работу или IO — это ответственность data layer.
- GUI получает события из data layer через `try_recv()` в каждом кадре (`eframe::App::update`),
  **до** сбора новых интентов: try_recv() → collect intents → handle_intent.
- Intent'ы, требующие работы data layer (Format, Validate, OpenFile, SaveFile), отправляются как `EditorCommand` в канал.
- **Горячие клавиши**: F5 — форматировать, F6 — проверить, Ctrl+O — открыть, Ctrl+S — сохранить,
  Ctrl+Shift+S — сохранить как. Обрабатываются через `consume_key` в `app.rs::update()`.
- **Подсветка синтаксиса**: через `TextEdit::layouter()` и `build_highlighted_job()`.
  Цвета токенов: G-коды салатовые, M-коды синие, оси жёлтые, AxisExpr тёмно-жёлтые,
  комментарии серые, WHILE/IF светло-бардовые, Word бардовые, Speed небесно-голубые,
  RParameter тёмно-синие.
- **Подсветка ошибок**: строки с ошибками — красный полупрозрачный фон.
- **Тестирование GUI**: через `egui_kittest` + `Harness`, проверка виджетов через AccessKit-дерево
  (поиск по тексту, роли, состоянию).


## Особенности G-кода (НЕ ошибки — не пытайся «исправить»)
- **Точка с запятой `;`** — всегда начало комментария до конца строки. Не зависит от контекста. Всё после `;` игнорируется.
- **Оси без G-кода на отдельных строках** — строки вроде ` X`, ` Z=71.304`, ` Y-58.346` без G-кода в начале — это переход к следующей команде движения. Они должны оставаться на отдельных строках, не нужно группировать/приклеивать их к предыдущей строке.
- **`AxisExpr` (ось + `=` + выражение)** — конструкции вида `Z=71.304`, `X=160+10`, `Y=3*5/2` **неприкосновенны**. Никакого округления, пересчёта или изменения выражения. Парсер преобразует их в `Statement::Word`.
- **`Speed` (скорость шпинделя)** — `S1000`, `S1=1000`, `SS1=500` — отдельный токен `Token::Speed`, не изменяется.
- **`RParameter`** — `R50`, `R101=R101+1` — отдельный токен `Token::RParameter` (только R + цифры,
  команды вида `RCHECK` — это `Token::Word`).
- **`decimal_places`** — у `Token::Axis` есть поле `Option<usize>` — количество знаков после запятой.
  При форматировании незначащие нули отбрасываются: `X10.000` → `X10`, `X10.100` → `X10.1`.
- **Многосимвольные команды** — `MODECHECK(2)`, `TRANS`, `MATLCH("...")`, `CFTCP`, `MAMILL`,
  `WGTRANS(1)`, `MSG("...")`, `MATLRET` и другие — обрабатываются как `Token::Word`, не изменяются.
- **Управляющие конструкции** — `WHILE`, `IF`, `ELSE`, `ENDIF`, `ENDWHILE`, `REPEAT`, `UNTIL` —
  обрабатываются как ключевые слова, формируют блоки `WhileBlock` / `IfBlock` с телом и отступами.
- **Скобки `(...)` не комментарий** — выражения в скобках это параметры команд, а не комментарии. Только `;` начинает комментарий.
- **Пустые строки сохраняются** — если в исходном коде есть пустая строка, она должна остаться и в отформатированном выводе.
  При перенумерации со `skip_empty_lines=true` множественные пустые строки схлопываются в одну.
- **Перенумерация кадров** — **пост-обработка**. `format_block` форматирует код без перенумерации.
  Затем `apply_renumbering()` проходит по строкам и добавляет N-коды последовательно с заданным шагом.
  Комментарии (`;...`) не нумеруются. Строки только со старым N-кодом удаляются (счётчик увеличивается).



## Пример ожидаемого ответа
[План действий]: Создам сущность пользователя в domain слое
Архитектурное место: `src/domain/user.rs` — ядро бизнес-логики
```rust
//! Модуль доменной сущности пользователя

use chrono::{DateTime, Utc};

/// Пользователь системы с валидацией email
#[derive(Debug, Clone, PartialEq)]
pub struct User {
    pub id: i32,
    pub email: String,
    pub created_at: DateTime<Utc>,
}

impl User {
    /// Создаёт нового пользователя после проверки email
    pub fn new(id: i32, email: String) -> Result<Self, String> {
        if email.contains('@') {
            Ok(Self { id, email, created_at: Utc::now() })
        } else {
            Err("Неверный email".to_string())
        }
    }
}
